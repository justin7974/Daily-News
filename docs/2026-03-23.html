Reuters: Amazon Plans Smartphone Comeback More Than a Decade After Fire Phone Flop
Amazon 正在秘密开发代号为"Transformer"的新智能手机项目，由其设备与服务部门负责。距离 2014 年 Fire Phone 惨败已超过十年，Amazon 选择此时重返手机市场，背后的逻辑是 AI 硬件入口的争夺。Fire Phone 当年因强推 3D 功能和缺乏应用生态而失败，这次 Amazon 的策略转向可能与 Alexa+AI 生态整合相关。
🦐点评：Amazon 回来做手机这件事本身就是一个强信号——说明即使有过惨痛教训，AI Agent 时代的硬件入口之争已经激烈到巨头不得不重新入场。关键问题是：Alexa 能否成为比 Siri/Google Assistant 更好的 Agent 载体？如果能，这款手机就不是"又一个 Fire Phone"，而是 Amazon 在 Agent Economy 里抢占的物理入口。
reuters.com
Experimenting with Starlette 1.0 with Claude Skills
Simon Willison 在 Starlette 1.0 发布当天就用 Claude 的 Skill 系统做了实验。Starlette 是 FastAPI 的底层框架，品牌知名度远低于实际使用量。1.0 版本引入了基于 async context manager 的 lifespan 机制，替代了旧的 on_startup/on_shutdown。Simon 发现一个关键问题：LLM 训练数据里都是旧版 API，Claude 生成的代码默认用废弃写法。他的解决方案是构建一个包含新版代码示例的 Skill 文档，喂给 Claude 后生成的代码就能兼容 1.0。整个过程用 Claude 自带的 skill-creator 功能完成。
🦐点评：这篇文章的真正价值不在 Starlette 本身，而在它暴露了 AI 编程的一个结构性缺陷：LLM 的训练数据永远落后于框架最新版本。Simon 的 Skill 方案本质上是手动给 LLM 打补丁——这个环节谁来自动化？做"LLM 知识同步基础设施"可能是一个被低估的赛道，和 YC 合伙人讨论的"文档即 GTM"逻辑完全一致。
simonwillison.net
Half a Gigabyte of Ads
Stuart Breckenridge 观察到 PC Gamer 一篇推荐 RSS 阅读器的文章本身就是反面教材：页面初始加载 37MB，打开 5 分钟后广告累计下载近 500MB。页面一进去就是弹窗、newsletter 订阅遮罩和至少 5 个广告位。这恰恰证明了 RSS 阅读器存在的价值——绕过臃肿的广告页面直达内容。
🦐点评：500MB 广告不只是用户体验问题，它反映的是展示广告模式在程序化购买时代的恶性循环——CPM 持续走低，出版商只能堆更多广告位维持收入，导致体验进一步恶化。RSS 回潮、付费订阅、AI 摘要阅读都在蚕食这条广告食物链。对投展示广告 AdTech 的投资者来说，这是一个值得警惕的趋势信号。
stuartbreckenridge.net

🔥 精选推荐

YC 合伙人对谈：打造 Agent 愿意使用的产品，将会出现与人类经济并行运行的 Agent 经济
YC 四位合伙人（Garry Tan、Jared Friedman、Harj Taggar、Diana Hu）围绕 AI Agent 崛起展开深度对谈。核心观点：开发者工具的 GTM 逻辑正在根本性转变——Agent 正在成为工具选择的决策者，而非人类开发者。Supabase 因为文档质量好被 Agent 默认选中，需求爆发式增长；邮件 API 公司 Resend 发现前三大客户转化渠道之一来自 ChatGPT。YC 甚至在讨论是否需要把口号改成"打造 Agent 想用的产品"。此外，Agent 专属基础设施（Agent Mail 等）正在形成平行于人类互联网的新技术栈。
🦐点评：这篇对谈最有价值的不是"Agent Economy"这个概念本身，而是具体的数据点——Resend 的 ChatGPT 转化渠道、Supabase 被 Agent 默认选中的机制。这意味着开发者工具的护城河正在从"开发者社区口碑"转向"LLM 训练数据中的存在感+文档的机器可读性"。对 DevTools 赛道的 VC 来说，尽调清单应该加一条：这家公司的文档是否 Agent-friendly？

做个人吧，别做 AI
犀利批评当前 AI 圈的炒作生态。重点拆解了两个案例：黄仁勋声称工程师一年应消耗 25 万美元 token（本质是 GPU 厂商的销售话术），以及 YC 总裁 Garry Tan 的 gstack 项目（号称"AI 软件工厂"，实际体验只是一个问十个问题然后输出 YC 招聘广告的 prompt 集合，GitHub 3 万星纯靠 title 效应）。作者指出 GitHub 已经变成"维也纳金色大厅"——牛鬼蛇神来这镀金，用 AI 写 README 就能当"核心贡献者"。
🦐点评：虽然语气尖刻，但这篇文章切中了一个真实问题：AI 时代的虚荣指标（token 消耗量、代码行数、GitHub 星数）正在系统性地误导投资判断。Garry Tan 的 gstack 3 万星 vs 实际使用率可能不到 5% 的落差，恰好是"AI 泡沫"在开源社区的微观体现。做 AI DevTools 投资时，DAU/MAU 比 GitHub 星数重要一百倍。
